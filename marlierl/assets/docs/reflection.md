## How do your procedures address the needs of a "reading to learn to do audience"? Provide some specific examples that connect back to Redish's noted features of such an audience.

the written procedure not only explains an action (reading to do) but the procedure is written clearly and concisely enough that when the user reads it they can retain the knowledge (reading to learn) and hopefully will not have to use the procedure again because they will know what to do next time.



## How do your procedures follow the WTGA Staging, Coaching, and Alerting stylistic conventions? Provide some specific examples that connect back to the criteria from Hart-Davidson.

Before each procedure, I included a warning to alert the user that there are specific prerequisties that need to happen in order to be successful.

Almost all coaching steps begin with an action-verb.

I broke up my coaching actions into stages to help guide the user and make it easier for non-novice users that may just be returning the procedure for a refresher - that way they can easily identify which section of the procedure they need to look at.



## How do your procedures follow your task orientation work? In other words, based on your audience and their goals, discuss how you oriented your SCA moves to the tasks.

Doing the task orientation work prior to starting to write the actual procedure was extremely helpful. I had the task analysis doc up on my computer the whole time was writing the code and constantly referred back to it. That said, I also made some changes to make the tasks and user scenarios to make them more useful for the actual written procedure. It was like the task analysis was the rough draft and the written code was the final draft.
